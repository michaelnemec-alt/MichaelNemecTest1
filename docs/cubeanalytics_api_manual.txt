CubeAnalytics API documentation
This page explains key concepts and provides the documentation for the CubeAnalytics API.

The API documentation listed below contains information on how to use the API, how the API is structured, and what information you can find in the API.

In the AutoStore system, there are some common metrics and terminology. Two of these detailed in Concepts: describing how we calculate different metrics.

If you have any questions regarding the API and/or documentation, reach out to your AutoStore representative.

We continuously expand and evolve the product CubeAnalytics. Change logs are described clear and concise, and have email notifications available. Toggle on/off in the section in the upper right corner.

Getting started
Authentication
All API requests require an authentication token, which can be managed at /api-tokens. To use the token, include it as an HTTP header in your requests:

API-Authorization: Token MY_TOKEN

Examples
List Installations Example
Querying with curl

curl -s -H "API-Authorization: Token MY_TOKEN" "https://api.cubeanalytics.autostoresystem.com/v1/installations/"
Querying with Postman

Method: GET
URL: https://api.cubeanalytics.autostoresystem.com/v1/installations/
Headers:
Key: API-Authorization, Value: Token MY_TOKEN

REST API overview (Historical data)
These endpoints enable users to retrieve and analyze historical data for various aspects of their AutoStore installation, offering insights into performance, efficiency, and operational health. This data is updated daily.

List Installations (/installations): Lists the installations a user has access to, including unique IDs and site names.
Status: Returns the status of the grid.
Access Point Load (/access-point-load): Offers day-aggregated data related to radio traffic and packet loss for modules’ access points.
Bins Content Code (/bins-content-code): Analyzes the number of bins per content code, indicating fill rates and empty bin statuses.
Bin Quality (/bin-quality): Provides data to assess if robot errors are related to the quality of bins, including bin IDs connected to robots pre-error.
Bins Above (/bins-above): Offers data on the number of bins above a given bin, aiding in understanding digging depth and task distribution.
Bin Rescue (/bin-rescue): Provides data on potential bin rescues, including robot IDs, error types, and estimated downtime.
Grid Info (/grid-info): Shows different capacities in bins and cells across grid types.
Grid Layout (/grid-layout): Describes the grid structure, detailing cell types and modes for constructing a visual grid representation.
Incidents (/incidents): Contains incident data, including incident IDs, statuses, and details.
Event Log (/event-log): Contains event log data akin to what is seen in the AutoStore Console.
Installation Data (/installation-data): Summarizes installation data, covering bins, robots, and ports.
Module Versions (/module-versions): Summarize the versions of various modules within the installation.
Live Events (/live-events): Deprecated. Use Live Events Stream instead. Live data events that can be retrieved by date, last hours or sequence ID.
Live Events Stream (/live-events-stream): Live data events that can be retrieved by date, last hours or sequence ID. (Recommended)
Robot Errors (/robot-errors): Provides data on robot errors, including error codes and timestamps.
Robot Mean Time Between Failures (/robot-mtbf): Provides extensive data on robot performance, including mean time between failures.
Robot Movement (/robot-movement): Offers data on robots' movements.
Robot State (/robot-state): Returns which states the robots has been in during the specified timespan.
Uptime (/uptime): Provides data needed to calculate the installation's uptime, a crucial performance metric.
Port Bin Wait Time (/port-bin-wait-time): Gives a summary of bin waiting times at ports throughout the day.
Port MBBF (/port-mbbf): Delivers data on port errors and bin throughput, calculating mean bins before failure.
Port State (/port-state): Provide data of which different states the port has been in.
Port Uptime (/port-uptime): Details port statuses (up, closed, down, stopped) and the errors contributing to downtime.
Port Pick Speed Distribution (/port-pick-speed-distribution): Analyzes time distribution for bin presentations at ports, aiding in operational efficiency.
R5 Robot Battery (/R5-battery): Tracks battery data for R5 robots, including charging times.
R5 Battery Estimation (/r5-battery-estimation): Provides battery estimation data for R5 robots.
R5 Chargers (/R5-chargers): Offers data on R5 chargers and robot interactions with them throughout the day.
R5.1 Battery (/r5-1-battery): Provides data from charging on R5.1 and R5.1 Pro robots.
R5.1 Charger (/r5-1-charger): Provides data from charging on R5.1 and R5.1 Pro chargers.
Bin Presentations (/bin-presentations): Provides data on bin presentations for goods in or picks per bin each day.
B1 Battery (/B1-battery): Details battery data for B1 robots' charging activities.
B1 Chargers (/B1-chargers): Provides charger data for B1 robots, including usage throughout the day.
Bin Presentations Trend (/bin-presentations/trend)
Uptime Trend (/uptime/trend)
Robot Outlier (/robot-outlier): Identifies robots that behave as outliers compared to their peers, both in battery and operation.
Port Outlier (/port-outlier): Identifies ports that behave as outliers compared to their peers.
AI Summary (/ai-summary): Provides an AI-generated daily summary of the installation's health, reliability and performance.
System Health (/system-health): Provides an overall health index for the installation along with KPI scores and supporting metrics.
WebSocket overview (Live data)
CubeAnalytics provides a WebSocket-based API for live data streaming. This real-time data is essential for immediate insights into the operations of your AutoStore installation. Harness these for instant alerts, to empower dynamic decision-making, or to feed live operational data into your custom-built management tools.

Prerequisites for Access

You need Unify Connect version 1.6.3 or higher.
Live data must be enabled for your installation by our team.
Available Events

SYSTEM_MODE: Indicates changes in system operation mode instantaneously.
ROBOT_STATE: Sends the aggregate and current state of the robot every 5 minutes.
ROBOT_ERROR: Reports errors occurring on any robot instantaneously.
PORT_ERROR: Reports errors occurring on any port instantaneously.
BIN_AND_TASK: Sends aggregate of bin and task-related data every 5 minutes.
PORT_STATE: Sends the aggregate and current state of the ports every 5 minutes.
DOOR_STATE: Indicates the state of grid or robot doors upon changes instantaneously.
INCIDENT: Reports incidents occurring in the system instantaneously. Requires Unify Connect version 1.7.0 or higher.
DELAYED_SYSTEM_STOP: Reports when a delayed system stop has been activated and when it timeouts. Requires Unify Connect version 1.8.0 or higher.
CHARGER_STATE: Sends the aggregate and current state of the chargers every 5 min. Requires Unify Connect version 1.9.0 or higher.
STATUS (Live): Sends the current mode of the grid and the client's connection status to Driver Requires Unify Connect version 1.12.0 or higher.
Please see the WebSocket API documentation for more details.

Use Cases
Automated Error Response System
Scenario: Service Engineers need prompt responses to system errors.

Application: Implement a system to monitor SYSTEM_MODE events. Automatically initiate corrective actions or notify technicians via your organization’s communication platform.

Custom Alerts for Operational Thresholds
Scenario: Operation Supervisors require setting thresholds for key operational metrics.

Application: Use Port Bin Wait Time and Access Point Load metrics to establish custom alerts. These alerts activate when wait times surpass predefined acceptable limits

Workforce Planning
Scenario: Operation Managers need to strategize workforce shifts based on activity levels.

Application: Analyze Port Pick Speed Distribution and BIN_AND_TASK data to identify peak times. Utilize this data for strategic shift scheduling and efficient human resource allocation.

Custom Reports for System Health and Performance
Scenario: Stakeholders demand regular insights into system health and performance.

Application: Develop a script utilizing multiple API endpoints for a detailed health report of the AutoStore system. Automate this to generate daily or weekly.

Integrate AutoStore data with other sources
Scenario: Enhancing system efficiency through integration with external systems.

Application: Integrate AutoStore’s system data with external data sources like conveyors, sortation systems, WMS, and robotic picking. This integration facilitates a more cohesive reporting.

Overview and design philosophy
The REST API focused on an installation resource. Given an installation ID, the user can query different data for this installation and expect to get the data in a certain structure. The data is returned as JSON. All requests to the API are HTTP GET requests.

Endpoints where my-installation-id is required, you can only query data for a single installation per query, meaning you have to make seperate requests for separate installations. This is due to amount of data that can be transferred per request. To fetch data from multiple installations, make multiple queries either in sequence or in parallel.

The API provides a set of default filtering options that can be used on each endpoint to adjust the result. The filtering options are:

after: only get results after this date. Accepts either a date in the format YYYY-MM-DD or "latest" to fetch the most recent available data.
before: only get results before this date.
version: only get results start with this version number.
Both the after and before filter are inclusive on the date provided. Sorting is done by providing the latest day first in the results. If there are no results available, the results array will be empty. Filtering on the API is optional.

The response from the API follows a certain structure. First, the result data is paginated, meaning that if the size of the result data goes above a certain threshold, more queries need to be made to get the rest of the data. The result from an endpoint has the following structure:

{
  "next": "https://url.to.next.page",
  "previous": null,
  "results": [
    {
      "version": "1.0.0",
      "date" : "2021-01-01",
      "result": DATA_FOLLOWING_SCHEMA,
    },
    {
      "version": "1.0.0",
      "date" : "2021-01-02",
      "result": DATA_FOLLOWING_SCHEMA,
    },
    ...
  ]
}
The results key contains a list of all result items. Each result item is the result for a given date. A result for a date follows a specific schema. The schema for each endpoint and version is described in the endpoints section below.

For each endpoint, JSON Schema is used to describe the data available at the endpoint. Only the schema for the latest version of the endpoint is currently provided.

For access to the API, contact your AutoStore Partner.

Concepts
The CubeAnalytics API provides metrics and key data relevant for the performance, uptime and throughput of your AutoStore installation, and how we calculate these values might not be straight forward or intuitive.

However the following documents aim to provide more details on how we calculate the most common metrics: Uptime concept and Bin Processing Times

List installations
GET v1/installations/
Description
Installations that the user has access to are listed in this endpoint. This endpoint does not allow for filtering query parameters.

Are you missing any installations in the list retrieved from the server? Then you probably need to synchronize available installations in CubeAnalytics.

You do this in the CubeAnalytics web app by clicking your user icon and select Synchronize.

Access point load
GET v1/installations/<my-installation-id>/access-point-load
Description
The packet loss endpoints contains day-aggregated data for packets (radio traffic) for the access points at modules. It can be used to calculate metrics such as packet loss ratio, which gives an indication of the radio performance on the installation. v2 provides the load data in an hourly interval.

Bins content code
GET v1/installations/<my-installation-id>/bins-content-code
Description
The bins content code analysis gives the amount of bins per content code, and if that content code indicates empty bins or not. It can be used to calculate metrics such as fill rate.

Bins above
GET v1/installations/<my-installation-id>/bins-above
Description
Bins above endpoint provide information of how many bins are above a given bin, or the digging depth when getting that bin. When a bin is in a port or prepared, bins above is 0. The first time a bins is requested it will be N number of bins above, afterwards it assumes that it will be prepared and will be 0 bins above. Otherwise, bins above will be number of bins located above the specific bin. The digging depth is the number of bins that needs to be moved to get the bin in question. This is the same as the number of bins above the bin in question.

Bin Rescue
GET v1/installations/<my-installation-id>/bin-rescue
Description
The bin rescue endpoint provides data on potential bin rescues, including robot IDs, bin IDs, error types, estimated system downtime reduced, position of the error and if the error stopped the system. It also include cell downtime reduced. It can be used to see how much system downtime and cell downtime the bin rescue robot could avoid. This is an estimation and is estimated by looking at the how long the system downtime and cell downtime event lasted and assuming that the bin rescue robot would have avoided the system downtime and cell downtime.

Description
The grid-layout endpoint represents the grid as a json structure. The data goes over all the cells in the grid and provides the cell TYPE and MODE for each cell. From this data, it is possible to construct a representation of the grid similar to that in ASConsole. Incidents

Description
The incidents endpoint provides data on all historical incidents that have occured at the installation. It provides information on when an incident occured and when it was resolved, if resolved. If an incident is still active from the previous day, it will be included in the next day's data.

Live events
Deprecated. This endpoint is deprecated and may be removed in a future version. Use the Live events stream endpoint instead, which provides the same functionality.

GET v1/installations/<my-installation-id>/live-events
Description
The live-events endpoint (deprecated) houses a repository of events received via the WebSocket connection. There are two methods available for retrieving these stored live events. The first approach entails employing a date metric either with the "after" and "before" parameters (allowing for a maximum time span of three days), or with the "lastHours" parameter (allowing a maximum of 48 hours). For instance:

Example 1:

v1/installations/<my-installation-id>/live-events?after=<yyyy-mm-dd>&before=<yyyy-mm-dd>
Example 2:

v1/installations/<my-installation-id>/live-events?lastHours=<int>
The alternative option involves using the "connection_id," "sequence_number_from," and "sequence_number_to" parameters. Here's an example:

v1/installations/<my-installation-id>/live-events?connection_id=<id>&sequence_number_from=<int>&sequence_number_to=<int>
Both methods return a list of events in adherence to the specified parameters, utilizing the same schema. Notably, Example 2 also contains a list of missing sequence numbers based on the provided parameters located at the final index of the list of events.

There is also possible to filter on the different events by adding the query parameter "eventType". No specification will return events of all types

Example:

v1/installations/<my-installation-id>/live-events?after=<yyyy-mm-dd>&before=<yyyy-mm-dd>&eventType=<event_type>

GET v1/installations/<my-installation-id>/live-events-stream
Description
The live-events-stream endpoint houses a repository of events received via the WebSocket connection. There are two methods available for retrieving these stored live events.

The first method employs a date-based metric and can be utilized in one of two ways: either with the "after" and "before" parameters, (allowing for a maximum time span of three days) or with the "lastHours" parameter, which permits retrieval of events from the past 48 hours.For instance:

Example 1:

v1/installations/<my-installation-id>/live-events-stream?after=<yyyy-mm-dd>&before=<yyyy-mm-dd>
Example 2:

v1/installations/<my-installation-id>/live-events-stream?lastHours=<int>
The second method involves "connection_id" together with "sequence_number_from" and "sequence_number_to" parameters to fetch events within a specific sequence range. Here's an example:

v1/installations/<my-installation-id>/live-events-stream?connection_id=<id>&sequence_number_from=<int>&sequence_number_to=<int>
Both methods return a list of events in adherence to the specified parameters, utilizing the same schema. Notably, Example 3 also contains a list of missing sequence numbers based on the provided parameters located at the final index of the list of events.

There is also possible to filter on the different events by adding the query parameter "eventType". No specification will return events of all types

Example:

v1/installations/<my-installation-id>/live-events-stream?after=<yyyy-mm-dd>&before=<yyyy-mm-dd>&eventType=<event_type>

Robot errors
GET v1/installations/<my-installation-id>/robot-errors
Description
The robot errors endpoint provides data on all historical robot errors that have occured at the installation. It provides information on when an error occured and where it occured if available. It also provides information on the error type and error code and if the error caused the system to stop or a delayed stop. It include more errors than the MTBF endpoint since it includes all errors and not only the ones used to calculate the MTBF. This endpoint might differ from what the UPTIME endpoint reports causing system stop because ROBOT ERRORS reports the root cause of the system stop while UPTIME reports the error that actually caused the system stop. This might occur when a robot receives a new error during recovery which the system is not able to recover from.

Robot Mean Time Between Failures
GET v1/installations/<my-installation-id>/robot-mtbf
Description
Robot MTBF (mean time between failure) is an important metric for calculating the performance and durability of the robots. The Robot MTBF data is extensive and the data the endpoint provides can also be used to calculate other things such as the errors or active times of the robots. Only robot errors such as errors happening when the is active and operational are included in the MTBF calculation. Errors happening during recovery are also excluded.

NOTE: Not all errors are included in the MTBF calculation. The include_error field in Robot Errors (/robot-errors) will inform if the error is included in the calculation. This is done to filter out errors that are not relevant for the MTBF calculation.

Robot State
GET v1/installations/<my-installation-id>/robot-state
Description
The Robot State endpoint gives data categorizing each robot into 8 different states: AVAILABLE, UNAVAILABLE, WORKING, CHARGING_AVAILABLE, CHARGING_UNAVAILABLE, SERVICE_ON_GRID, SERVICE_OFF_GRID, or RECOVERY. The data is aggregated per hour. The patternProperties is a regular expression that matches the format of the timestamp. The timestamp is the hour of the day in local time of the installation. Example: 00:00:00 is the first hour of the day, 01:00:00 is the second hour of the day, and so on.

Uptime
GET v1/installations/<my-installation-id>/uptime
Description
The uptime endpoint gives all data required to calculate the defined uptime for the installation. Uptime is one of the most important metrics for calculating the performance of a site. To calculate the uptime, recovery mode must be used and understood.

Changelog
1.6.0 / 2026-05-20
Uptime analysis response schema aligned with v1.6 output; fields, enums, and formulas are in the OpenAPI block above.
Prefer structured system-stop fields over parsing legacy stop_text (unchanged, still populated). robots_error_at_system_stop is documented above for downtime periods only.
1.5.1 / 2026-04-09
Uptime analysis will no longer consider daylight and timezone changes as missing time.
1.5.0 / 2026-01-05
Added new properties of "response_seconds", both total of the day and of each period.
1.4.2 / 2025-05-08
Fixed a bug retrieving the correct stop data from the log when the system is down at the beginning of a new day, and there are multiple system start attempts during the downtime period.
1.4.1 / 2024-05-31
Fixed a bug retrieving the correct stop code from the log in certain manual stop edge cases. We will now get the stop code up to and including the first system mode STOPPED segment of the downtime period.
1.4.0 / 2024-03-12
Added the periods where the log is missing. This will make more visible where there are gaps in the log. No changes were made to the rest of the calculations of the uptime analysis.
Added a new period type of 'mode' "missing".
Added 'missing_seconds' as the missing log seconds of the period.
Added 'missing_seconds' as the total amount of seconds missing of each day.
1.3.0 / 2024-02-27
Added 'stop_code' as the value of the 'stop_code_name'.
1.2.4 / 2023-12-22
Fixed bug where 'recovery_at' was not being set correctly across days causing 'recovery_seconds' also being incorrect.
1.2.3 / 2023-11-14
Fixed bug where stops were not getting the correct stop data. Some robot stops were being counted as manual stops.
Add system mode ALERT as an 'uptime' system mode. Now system modes RUNNING, SERVICE and ALERT are considered as 'uptime'.
1.2.2 / 2023-11-07
Fixed bug where manual stops were not being accounted for correctly. Both 'STOPPED_FROM_CONSOLE' and 'GRID_DOOR_STOP' are manual stops.
1.2.0 / 2023-05-11
Added value 'STOPPED_FROM_CONSOLE' to stop_code_name field to indicate a manual stop.
Added manual_stop_reason field.
1.1.2 / 2023-05-08
1.1.1 / 2023-04-19
Recovery seconds of manual stops periods now start counting from when the grid door is opened, that is, the same behaviour as all the other stops.
Allows counting of time spent waiting for superuser to open the grid door.
Recovery seconds of manual stops are no longer counted for total recovery_seconds.
Since recovery_up_ratio uses this value, the uptime calculation is updated.
1.1.0 / 2023-10-01
Added property stop_code_name.

First version
Port MBBF
GET v1/installations/<my-installation-id>/port-mbbf
Description
The Port MBBF (Mean Bins Before Failure) endpoint provides the number of port errors as well as the number of bins that went through the ports in the installation throughout the day. The "Mean bins before failure" value gives us an average of the number of bins that go through a port before we have an error in the port.

Port State
GET v1/installations/<my-installation-id>/port-state
Description
The Port State endpoint provides data on how long a port was in different states for each hour during a day. This endpoint can be used to see the port availability and uptime. It also can be used to see for how long the port was closed or open. The different states are OPEN, CLOSED, DOWN, and STOPPED.The OPEN state is when the port can be used for pick or goods in. The CLOSED state is when the port is not in use. The DOWN state is when a port has received an error and is not operational. The STOPPED state is when the port has received an error and is not operational, but the error is defined as a user error. When the version includes "-p", it means that this is preliminary data and the complete data will be available in the next day. This is because we receive data once a day and in some cases required events happens the next day.

Port uptime

Description
The Port Uptime endpoint provides the time in where each port was up, closed, down or stopped in the installation throughout the day. The list of periods allows us to know which errors were responsible for the downtime or stop of the ports. When the version includes "-p", it means that this is preliminary data and the complete data will be available in the next day. This is because we receive data once a day and in some cases required events happens the next day.

Port Pick Speed Distribution
GET v1/installations/<my-installation-id>/port-pick-speed-distribution
Description
The Port Pick Speed Distribution endpoint gives time distrobution across different categories of times for each bin presentation for each port in an installation. The data helps in understanding how long the AutoStore system uses to provide a bin, how long operators use to handle the bin, and how long the Warehouse Management System (WMS) uses to give AutoStore new tasks to complete, presented as frequency distributions. This endpoint is useful for operational analytics, enabling fine-grained performance measurement down to the hour and port level. Use it to optimize queue lengths, staffing, and WMS configurations.

WebSocket details (Live data)
The CubeAnalytics server provides a websocket-based API for live data. This data is delivered in events which are published to the listeners on the websocket. Please note that only installations that have a Unify Connect version greater than 1.5.0 can access live data. Additionally, live data must be enabled for the installation by AutoStore. Contact your AutoStore representative to enable live data for your installation. The AutoStore Demo site contains a preview of what live data looks like.

Version Overview
Below is an overview over what events (including version) each Unify Connect versions send.

Unify Connect 1.12.0
Unify Connect 1.11.5 / 1.11.7
Unify Connect 1.10.0
Unify Connect 1.9.0
Unify Connect 1.8.2 / Unify Connect 1.8.6
Unify Connect 1.7.0
Unify Connect 1.6.3
Connecting to the WebSocket
To connect:

Make a GET request to https://live.cubeanalytics.autostoresystem.com/connect?installations=MY_INSTALLATION with your API token in the Authorization header "Authorization": "Token MY_TOKEN".
Use the response from step 1, a wss://cubeanalytics.webpubsub.azure.com URL with an acess_token, which is valid for 10 minutes, to connect to the websocket.
Please note:

Use 1 API-token per WebSocket connection. When a connection is made, all other active WebSocket connections using the same token will be disconnected. Therefore create a 1:1 number of tokens to servers where you want to connect.
/connect requests are rate limited to 100 request per 5 minutes per IP adress.
The wss:// URL is valid for 10 minutes, after which you need to request a new URL.
The permissions for the WebSocket connection are the same as set for the API token.
Without a ?installations= parameter, you will be connected to all installations the token has access to. If you want to connect to specific installations, add the ?installations=A,B,C parameter to the URL.

Once connected, the server will start pushing events to your application. Each event is a JSON object that represents a change in the system. See the Event section for JSON schema for these events.

Events
When connected to the WebSocket, you will receive set of JSON events. All the events follow a specific schema, and they are in turn described below. The events, and thus the corresponding schema, are given in the JSON format.

In general, each event has the following schema:

General schema
The possible event types are: SYSTEM_MODE, ROBOT_STATE, ROBOT_ERROR, BIN_AND_TASK, PORT_ERROR, PORT_RETRY, DOOR_STATE, PORT_STATE, INCIDENT and DELAYED_SYSTEM_STOP.

SYSTEM_MODE
The system mode event is sent when the system changes mode, typically between RUNNING and STOPPED. If the system stopped abruptly, the stop code is sent along with contextualized data about the stop.

ROBOT_ERROR
The robot error event is sent every time an error occurs on a robot or when an error has been cleared. It contains information on which robot the error occurred, what the error is, where the robot was when the error occurred, if it had a bin, if it was in a port. It also contains the error code of the error.

BIN_AND_TASK
The BIN_AND_TASK event is sent every 5th minute and contains information on how many tasks are in the system, how many tasks were added in the previous five minutes, how many tasks were picked in the previous five minutes etc. It also contains the aforementioned information on a port basis.

Version < 2.0.0 of the event only supports Task interface. With the release of Unify Connect 1.7.0 producing BIN_AND_TASK events with version 2.0.0, the event will also support Bin interface.

DOOR_STATE
The DOOR_STATE event happens every time there is a change in the state of either the grid door or robot door. For each event sent, the state of all doors are sent. The states are either OPEN or CLOSED. The position of the door is given in the X and Y position on the grid.

DELAYED_SYSTEM_STOP
The DELAYED_SYSTEM_STOP event is sent whenever an incident occurs which causes the system to have a delayed system stop. Delayed System Stop needs to be active for the site for it to be sent. It contains information on what caused the delayed system stop and when the system will timeout.
